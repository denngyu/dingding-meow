# 严重错误记录

## 2026-07-16：饮水记录框底部说明再次被固定高度裁切

### 现象

- 确认按钮能够显示，但按钮下方的 `Enter 记录 · Esc 或右上 × 关闭误判` 提示被窗口底边裁掉。
- 无边框弹窗没有可拖动区域，用户无法自行移开或检查被遮挡部分。

### 根因与修复

- 新增四张快捷量卡片后的内容高度继续依赖固定 392px，底部没有留安全间距。
- 弹窗增高到 420px，并给背景、标题区与底部提示绑定窗口拖动；交互控件不绑定拖动。
- 固定高度弹窗的测试下限同步提高到 420px，后续增加文案或间距时必须重新核对最底部控件。

## 2026-07-16：橘猫邻格碎片与摄像头关闭状态混同遮挡

### 现象

- 暖橘皮肤的离座姿态右侧显示一条不属于当前猫咪的橘色碎片。
- 摄像头被硬件开关关闭或设备停止出帧时，界面仍可能显示“镜头被挡住了”，没有进入可点击恢复的停止检测状态。

### 根因

- `assets/skins/orange/cat_sprites/away.png` 裁切时带入了相邻图格边缘；原拓扑测试只覆盖经典皮肤。
- 检测循环只用画面均值判断遮挡，没有区分设备读取失败、数字纯黑无信号帧和普通低照度画面。

### 修复与预防

- 所有皮肤精灵在最终 144×144 显示尺寸上只保留最大 alpha 连通块，并为暖橘离座姿态补回归测试。
- 连续读取失败 8 次或连续纯黑无信号 3 帧后进入 `camera_off`，释放设备并显示“点击开始检测”；用户点击猫咪后才重启检测。
- 以后新增摄像头状态不得把“设备不可用”和“仍有图像但镜头遮挡”复用成同一个模式。

## 2026-07-16：橘猫睁眼时再次短暂闪出带尾巴坐姿

### 现象

- 切换到暖橘皮肤并展开监控画面后，小猫偶发闪成另一张带尾巴坐姿。
- 问题与之前经典皮肤的眨眼闪图相同，但只依赖通用状态测试，没有覆盖橘猫实际素材和最终绘制入口，因此再次被用户发现。

### 根因

- `blink` 周期仍然存在；虽然主状态选择已优先返回 `watch`，最终绘制入口没有独立校验“睁眼时禁止 `idle` / `tail`”这一视觉不变量。
- 项目根目录源码与 `dist` 发布目录不是同一运行副本；只验证源码不能证明开机自启动指向的旧 EXE 已包含修复。

### 修复

- 新增 `enforce_display_invariants()`，在 Canvas 最终取图前再次把睁眼状态下的 `idle` / `tail` 收敛为 `watch`。
- 增加暖橘皮肤实图回归测试，验证睁眼、眨眼两种状态都使用橘猫 `watch.png`，且不会误取 `idle.png` 或 `tail.png`。
- 增加无摄像头橘猫睁眼预览，连续跨过多个眨眼周期做启动级验收。

### 预防

- 新皮肤接入不能只检查图片完整和透明通道；所有跨帧状态必须使用真实皮肤素材跑一遍。
- 涉及“偶发闪图”的修复必须同时覆盖状态选择和最终绘制入口，不能只依赖单层分支。
- 交付时必须明确用户实际运行的是源码入口还是发布目录，不能把源码通过当作旧 EXE 已更新。

## 2026-07-16：未确认现场状态，把用户在座画面误标为空位

### 现象

- 用户报告离开后仍显示久坐。
- 诊断时没有先确认用户是否已经离开，就把摄像头检测结果称为“空位实测”。
- 随后基于错误标签加入局部运动活体过滤，导致静止用户无法被识别。

### 根因

- 把未知的现实环境状态当成已知测试前提。
- 自动化数据本身只说明“检测到稳定人脸”，不能证明画面主体是家具。
- 在没有正负样本标签的情况下直接调整召回阈值。

### 修复

- 撤销局部运动活体过滤和未验证的人体交叉确认，恢复原有真人识别链路。
- 保留与误判无关的检测线程 supervisor，避免线程异常时冻结旧状态。
- 文档中删除“空位 22 帧”和“家具误报已验证”等无效结论。

### 预防

- 依赖用户现实动作的测试，必须由用户明确告知“现在在座 / 现在离开”后才能标注。
- 未标注现场数据只能描述检测输出，禁止推断画面身份。
- 任何降低真人召回的过滤规则都必须先完成静止、侧脸、戴眼镜真人验证。

## 2026-07-16：静态家具修复误伤静止真人

### 现象

- 加入局部运动活体确认后，用户坐在摄像头前仍无法被识别。
- 实测 Haar 每帧都有正确人脸框，但活体层连续全部拒绝。

### 根因

- 把“必须出现可测局部运动”作为真人确认硬条件，忽略了用户可能长时间静止、
  帧间位移不足，以及眼镜会降低眼部分类器稳定性。
- 为修复家具误报引入了比原问题更严重的召回回归。

### 修复

- 撤销局部运动硬门槛及其后未经空位验证的人体交叉确认。
- 恢复原有真人识别链路，等待用户明确配合在座/离开两阶段复现。

### 预防

- 活体规则不能要求用户主动移动；检测准确率修复必须同时跑“空位”和“静止真人”实测。
- 任何负样本过滤上线前，都要验证戴眼镜、侧脸、静止三类真人召回。

## 2026-07-16：静态家具持续续命久坐状态（无效诊断，已撤销）

### 现象

- 用户离开座位后，小猫仍长期显示“久坐 1 小时”。
- 后续确认：所谓“空位摄像头实测”发生时用户仍在座位前，家具误报结论无效。

### 根因

- 未确认用户现场状态便把检测结果标成负样本，无法据此判断实际根因。
- 检测线程若异常退出，UI 缺少心跳保护，仍是一个独立的潜在冻结原因。

### 修复

- 删除基于无效样本实现的 `face_presence.py`。
- 新增检测线程 supervisor 与 UI 心跳保护，异常或停止更新时状态回落为离开。

### 预防

- 现实环境测试必须先取得可信的在座/离开标签，再讨论误报与漏报。

## 2026-07-14：图片猫接入后暴露三项核心使用缺陷

### 现象

- 饮水弹窗可以看到手填输入框，但底部确定按钮不可见，手填值无法正常提交
- 图片猫轮廓出现明显紫色描边
- 用户常用侧脸坐姿无法被识别为在座

### 根因

- 饮水弹窗固定高度只有286px，实际`pack()`内容更高，确定按钮被窗口边界裁掉；输入框也没有Enter提交绑定
- Windows Tk使用`#FE00FE`精确色键透明，PNG抗锯齿的半透明像素会先和紫色Canvas背景混合，形成无法被色键去掉的紫边
- 代码只加载`haarcascade_frontalface_default.xml`和`haarcascade_frontalface_alt.xml`；后者仍是正脸模型，项目复盘中写到的侧脸与镜像侧脸没有真正实现

### 影响

- 自动识别饮水后，用户无法可靠记录自定义毫升数
- 新角色素材在真实桌面环境下视觉破损
- 侧脸工作时会频繁依赖25秒宽限期，持续侧脸最终仍被误判离座，久坐统计失真

### 修复

- 饮水弹窗增高到376px，改用真实Button，绑定Enter和小键盘Enter；1–5000ml之外的输入原地报错
- 图片加载到最终144px尺寸后将alpha二值化，彻底移除色键透明产生的半透明紫边
- 新增独立`face_detection.py`，正脸失败后使用profile分类器检测左侧脸，并对镜像画面检测右侧脸后还原坐标

### 预防

- 固定尺寸Tk弹窗必须做控件底部与窗口高度的启动级检查
- 色键透明窗口的图片测试必须检查最终显示尺寸的alpha只包含0和255
- 文档声称支持某种检测角度时，自动测试必须验证对应分类器、镜像路径和坐标还原都实际存在

## 2026-07-14：眼镜被识别成玻璃杯

### 现象

- 用户佩戴眼镜时，杯子检测把眼镜当作玻璃杯，错误触发喝水动作和饮水记录弹窗

### 根因

- COCO类别集合同时启用了`bottle`、`wine glass`和`cup`；摄像头近景中的镜片与镜框容易被YOLO误判为第40类`wine glass`
- 原近脸规则只判断候选是否落在扩展人脸区域，没有排除眼睛高度的扁宽框，也没有限制候选的最低高度和纵向位置

### 影响

- 用户正常佩戴眼镜就可能频繁触发饮水流程，产生错误动画、弹窗和日志
- 单纯继续降低杯子置信度虽然能提高倾斜杯召回率，也会同步放大该误报

### 修复

- 饮水目标类别收紧为`bottle`和`cup`，移除高误报的`wine glass`
- 新增`is_cup_near_face()`几何过滤：排除眼睛区域、过矮和过宽候选，并约束杯子必须靠近嘴部到下脸区域
- 保留0.28候选阈值和倾斜杯允许比例，正常竖杯与轻微倾斜杯仍可通过

### 预防

- 杯子检测测试固定覆盖“眼镜扁框拒绝、远离下脸拒绝、正常杯通过、倾斜杯通过”四类场景
- 调整类别或阈值时必须同时验证误报和召回，不能只用单一杯子样例判断灵敏度

## 2026-07-15：启动验收误用全屏截图，带入项目外私人内容

### 现象

- 为验证中央叩屏位置，测试脚本截取了整块主屏，而不是盯盯喵自身窗口区域。
- 截图把桌面上其他应用的私人消息和敏感信息一并带入本地验收图及工具预览输出。

### 根因

- 把“确认小猫位于屏幕中央”错误实现成主屏全屏截图，没有先限定最小必要采集范围。
- 启动验收方案缺少“禁止捕获项目外桌面内容”的隐私检查。

### 影响

- 项目无关的私人内容被短暂写入工作记录截图，并进入本次工具预览上下文。
- 截图未加入 Git，发现后已立即从工作区删除。

### 修复

- 删除全屏验收截图，不将其提交或保留。
- 停止使用真实桌面截图作为分层透明 Tk 的验收手段；后续改用专用演示脚本的运行结果和离线 Canvas 渲染。

### 预防

- 桌面应用视觉验收默认只能截目标窗口或人工构造的空白测试区域；禁止全屏截图。
- 分层透明 Tk 即使窗口矩形正确，`CopyFromScreen` 也可能只捕获后景；不得据此认为截图内容局限在目标窗口。
- 需要视觉留档时只渲染人工构造的离线画布，不捕获真实桌面。
