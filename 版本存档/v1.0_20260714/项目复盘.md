## 一、需求是怎么来的

**起点**:手里有个摄像头,好奇"能拿它做点什么有趣的事"。

**第一个真问题**:"你(Claude)能不能看到我电脑的摄像头?"
- 探索后确认:Claude 能通过脚本抓帧 + 分析来"感知"画面(有几个人、亮度、光从哪来、有没有运动、甚至猫脸),但**不能实时盯着**(对话式,不是常驻进程)。
- 也就是说:真正 24 小时盯梢的得是脚本,Claude 负责写脚本、做关键判断。

**方向收敛**:脑暴过宠物看护、工位自律、延时摄影、安防等方向。最终选定 **久坐监督 + 延时摄影**,理由:
- 纯本地、免费、不用联网调 API
- 不涉及理解画面内容(那需要多模态 API),本地 OpenCV 就够
- 对小白友好、当场就能交付

## 四、踩过的坑(最有价值的部分)

### 技术坑

**1. opencv 5.0 删了 CascadeClassifier(最坑,绕了十几轮)**
- 现象:import cv2 成功,但 `cv2.CascadeClassifier` 不存在,人脸检测一初始化就崩。
- 误判:一度以为是"文件系统错乱 / 假 cv2 遮蔽 / 环境时好时坏",往玄学方向查了很久。
- 真因:opencv 5.0 是大版本升级,**删掉了老的 Haar 人脸检测 API**,换成了 FaceDetectorYN(DNN)。
- 解法:降级 `pip install opencv-python==4.10.0.84`,老代码一行不改就跑通。
- 教训:**遇到"玄学",先查最基础的东西 —— 版本号。**

**2. opencv 不认中文路径**
- 现象:延时摄影"拍了帧,却说没有帧可合成"。
- 真因:文件夹名"本地之眼"是中文,`cv2.imwrite / imread / VideoWriter` 对中文路径会**静默失败**(返回 False 不报错、文件没写成)。
- 解法:写用 `cv2.imencode + buf.tofile(path)`,读用 `cv2.imdecode(np.fromfile(path))`,视频先写系统 temp 的英文路径再挪回中文目录。

**3. 识别不准(托腮 / 偏头漏检)**
- 现象:手托着脸就检测不到,被误判"离开",还导致久坐计时清零、永远不提醒。
- 解法:多角度识别(正脸 + 侧脸 + 翻转侧脸)+ 暗光增强 + **宽限期**(偶尔漏检不算走,连续一段时间没人才算离座)。

**4. 捂住摄像头还说在座**
- 现象:捂住镜头,宽限期内还说"在座"。
- 解法:加**遮挡检测**,画面全黑(亮度极低)立刻判被捂、不吃宽限期。

**5. 摄像头独占** — 久坐监督和延时摄影不能同时开,会抢摄像头。

### 环境坑(这次调试环境特有的诡异)

- **双环境文件系统**:工具读写的文件系统和命令执行的不是同一个,Read 读不到 PowerShell 写的新文件。
- **文件时有时无 / 假 cv2**:文件写了又消失,一度冒出个删不掉的假 cv2 目录。
- **f 盘防护误报**:某些命令被"禁止删除 f 盘"拦截,但电脑上根本没有 f 盘。
- **exe 打包失败**:PyInstaller 打包 opencv 坑深(叠加 CascadeClassifier 缺失怎么打都白搭),果断放弃、改用 .bat。
- **输出缓存**:python 输出被 buffer,强制结束进程时丢失,看起来像"没跑",实际在跑 —— 要用 `python -u` 或 `print(flush=True)`。

## 五、经验教训

1. **先查版本,再信玄学** — 一整晚的"灵异事件",病根是一个 opencv 版本号。
2. **中文路径是 opencv 的雷区** — Windows 下 opencv 读写文件尽量避开中文路径,躲不开就用 imencode / imdecode 绕。
3. **小白工具:.bat 优于 exe** — 用户电脑已装 python 时,.bat 双击即用,绕开打包的所有坑。
4. **人脸检测要多角度 + 缓冲** — 单一正脸检测太脆,托腮、偏头就漏;加侧脸和宽限期才稳。
5. **验证要看真实输出** — 加 -u 无缓存,否则会被 buffer 骗、误判程序没在跑。
6. **区分"真坑"和"那一晚的鬼"** — 真正值得记一辈子的技术坑就两个(opencv 版本、中文路径);其余大半是那次调试环境自身的不稳定,换个干净环境就没有。
